# Аудит документации

Дата ревизии: 2026-06-30

## Цель

Этот документ фиксирует текущее состояние документации проекта и правила дальнейшей доработки. Он нужен, чтобы не переписывать материалы хаотично и не переносить в новые документы устаревшие утверждения.

## Текущее состояние

| Раздел | Статус | Комментарий |
|--------|--------|-------------|
| `docs/README.md` | Требует обновления | Есть навигация по старой структуре, но не хватает продуктового слоя, QA-сценариев и правил качества данных. |
| `docs/architecture/` | Частично актуально | Есть обзор, база данных, API-формат и деплой. Нужно сверить версии, маршруты, модели и фактическое поведение API. |
| `docs/modules/` | Частично актуально | Модули разложены по папкам, но часть статусов и описаний нуждается в проверке по коду. |
| `docs/data/` | Требует усиления | Есть JSON-данные, но не хватает документации по источникам, доказуемости и правилам обновления справочников. |
| `docs/qa/` | Недостаточно | Есть общий тест-план, но не хватает критических пользовательских сценариев и регрессионных чеклистов. |
| `docs_old/` | Архив | Использовать только как источник для миграции полезных фрагментов. Не считать актуальной документацией. |
| `reports/` | Справочно | Отчеты полезны для истории аудита питания и нагрузочных проверок, но не заменяют живую документацию. |

## Главные проблемы

1. В документации встречаются утверждения, которые нельзя доказать текущими файлами данных. Пример: полное покрытие всех пород или точное количество пород без отчета покрытия.
2. Описание модулей не всегда отделяет реализованную функциональность от планируемой.
3. Нет единого раздела для владельца проекта, партнера или инвестора: что делает сервис, какие сценарии закрывает, какие данные использует.
4. Нет единой методологии для источников пород, питания, здоровья и рекомендаций.
5. QA-документация не покрывает критические end-to-end сценарии: регистрация, PetID, подбор питания, корзина, checkout, оплата, отмена оплаты, напоминания.

## Правила ревизии

- Не переносить старый текст без проверки по коду.
- Не писать "все породы", "полная база" или "медицинская рекомендация", если это не подтверждено источником и отчетом покрытия.
- Все источники данных должны иметь ссылку, организацию, дату проверки и статус доверия.
- Каждый пользовательский сценарий должен иметь точку входа, результат и критерии приемки.
- Каждый технический документ должен ссылаться на реальные зоны кода или реальные API-маршруты.

## Приоритет доработки

| Приоритет | Документ | Причина |
|-----------|----------|---------|
| P0 | `docs/README.md` | Главный вход в документацию. |
| P0 | `docs/project/product-overview.md` | Нужен для понимания продукта без чтения кода. |
| P0 | `docs/project/user-flows.md` | Нужен для разработки, тестирования и демонстрации сценариев. |
| P0 | `docs/data/sources.md` | Нужен для доказуемости данных. |
| P0 | `docs/data/breeds/README.md` | Самый чувствительный справочник проекта. |

| P1 | `docs/architecture/frontend.md` | Нужно отдельно описать frontend-архитектуру. |
| P1 | `docs/architecture/backend.md` | Нужно отдельно описать backend-архитектуру. |
| P1 | `docs/security/` | Нужно описать доступы, роли, секреты, JWT и админки. |
| P1 | `docs/operations/` | Нужно собрать запуск, деплой, окружение и troubleshooting в один блок. |

## Готовность первой итерации

Первая итерация считается завершенной, когда есть:

- обновленный главный индекс документации;
- продуктовый обзор;
- карта пользовательских сценариев;
- методология источников данных;
- отдельная документация по справочнику пород;
- критический QA-чеклист.
