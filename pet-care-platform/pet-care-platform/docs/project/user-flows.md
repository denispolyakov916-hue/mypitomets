# User Flows

## Назначение

Документ описывает основные пользовательские сценарии Питомец+. Он нужен для разработки, тестирования, демонстрации продукта и постановки задач.

Каждый сценарий должен проверяться после крупных изменений frontend, backend, авторизации, корзины, оплаты, PetID и справочников.

## Роли

| Роль | Описание |
|------|----------|
| Гость | Пользователь без аккаунта. Может смотреть публичные страницы и начинать подбор питания. |
| Пользователь | Авторизованный владелец питомца. |
| Администратор | Staff/superuser пользователь с доступом к админ-панели. |

## UF-01. Регистрация через email

Цель: создать аккаунт и войти в сервис.

Точка входа:

- `/register`
- CTA после анонимной воронки подбора питания
- переход из защищенной страницы

Шаги:

1. Пользователь открывает регистрацию.
2. Вводит email, пароль и подтверждение пароля.
3. Система валидирует поля на клиенте.
4. Frontend отправляет запрос в backend.
5. Если аккаунт активируется кодом, пользователь вводит код.
6. После успеха пользователь попадает в целевой сценарий: PetID, корзина, рекомендации или профиль.

Ожидаемый результат:

- пользователь авторизован;
- токены сохранены корректно;
- нет зависшего loader;
- нет краша страницы;
- если был `returnTo`, пользователь возвращается в правильное место.

Критичные проверки:

- повторный submit не создает дубли;
- ошибка backend показана понятным текстом;
- пароль не логируется в консоль;
- Safari не показывает неожиданные системные окна из-за неправильной формы.

## UF-02. Вход через телефон

Цель: войти или зарегистрироваться по номеру телефона.

Точка входа:

- `/login`
- `/register`
- вкладка "Телефон" в `AuthModal`

Шаги:

1. Пользователь выбирает телефонный способ.
2. Вводит номер в читаемой маске.
3. Frontend нормализует номер для API.
4. Backend отправляет SMS-код или тестовый код в dev-режиме.
5. Пользователь вводит код.
6. Система авторизует пользователя и возвращает в целевой сценарий.

Ожидаемый результат:

- номер отображается удобно;
- в API уходит нормализованное значение;
- повторная отправка кода работает с cooldown;
- ошибки SMS/API не ломают страницу.

## UF-03. Создание PetID

Цель: создать цифровой профиль питомца.

Точка входа:

- `/pet-id`
- CTA "Добавить питомца"
- возврат из анонимной анкеты после регистрации

Шаги:

1. Пользователь выбирает вид животного.
2. Вводит имя.
3. Выбирает породу или отмечает беспородного/метиса.
4. Указывает дату рождения или возраст.
5. Указывает вес.
6. Заполняет пол, кастрацию/стерилизацию, активность и дополнительные поля.
7. Сохраняет профиль.

Ожидаемый результат:

- питомец появляется в списке;
- профиль открывается без перезагрузки страницы;
- все введенные поля сохранены в backend;
- незаполненные обязательные поля не подставляются автоматически неверными значениями.

Критичные проверки:

- собака не получает кошачьи породы;
- кошка не получает собачьи породы;
- дата рождения не сдвигается из-за UTC;
- десятичный вес корректно сохраняется;
- черновики не дублируются.

## UF-04. Редактирование профиля питомца

Цель: изменить данные питомца и увидеть сохраненный результат.

Точка входа:

- `/pet-id/:id`
- `/pets/:petId/edit`

Шаги:

1. Пользователь открывает карточку питомца.
2. Нажимает "Редактировать профиль".
3. Меняет активность, вес, питание, здоровье или другие поля.
4. Нажимает сохранить.
5. Закрывает редактор и открывает карточку снова.

Ожидаемый результат:

- backend возвращает успех;
- UI обновляется по фактическому ответу;
- после перезагрузки данные остаются;
- при ошибке сохранения пользователь видит точную причину.

## UF-05. Анонимный подбор питания

Цель: дать гостю ценность до регистрации и не потерять данные после авторизации.

Точка входа:

- `/start`
- `/pet-quiz`
- CTA на главной
- CTA из футера

Шаги:

1. Гость выбирает вид питомца.
2. Заполняет имя, породу, возраст, вес, кастрацию/стерилизацию, здоровье, бюджет и цель.
3. Получает страницу рекомендаций.
4. Нажимает "Добавить рацион в корзину" или "Сохранить".
5. Система отправляет гостя на регистрацию/вход.
6. После авторизации выполняется отложенное действие.

Ожидаемый результат:

- анкета не теряется;
- питомец или черновик создается ровно один раз, если сценарий этого требует;
- выбранный рацион не теряется;
- пользователь возвращается в корзину, PetID или рекомендации согласно действию;
- нет бесконечной загрузки.

## UF-06. Подбор питания для существующего питомца

Цель: авторизованный пользователь использует уже созданный PetID.

Точка входа:

- `/food-recommendation`
- `/recommendations`
- CTA на главной

Шаги:

1. Пользователь открывает подбор питания.
2. Система предлагает выбрать одного из существующих питомцев.
3. Пользователь выбирает питомца.
4. Система строит рекомендации на основе профиля.
5. Пользователь сохраняет рацион или добавляет товары в корзину.

Ожидаемый результат:

- не нужно повторно вводить уже известные данные;
- профиль питомца используется как источник персонализации;
- пользователь может уточнить недостающие поля.

## UF-07. Магазин и фильтры

Цель: найти товар и добавить его в корзину.

Точка входа:

- `/shop`
- категории в хедере;
- категории в футере;
- плашки магазина на главной;
- рекомендации питания.

Шаги:

1. Пользователь открывает магазин.
2. Выбирает категорию, вид животного, питомца, бренд, цену или другие фильтры.
3. Открывает карточку товара.
4. Добавляет товар в корзину.

Ожидаемый результат:

- URL содержит фильтры;
- перезагрузка сохраняет состояние фильтра;
- все категории открывают нужную выдачу;
- карточки и CTA кликабельны;
- корзина обновляется без полного мерцания страницы.

## UF-08. Checkout и заказ

Цель: оформить покупку.

Точка входа:

- `/cart`
- `/checkout`

Шаги:

1. Пользователь открывает корзину.
2. Выбирает товары/курсы.
3. Переходит в checkout.
4. Заполняет доставку, адрес и согласия.
5. Выбирает способ оплаты.
6. Создает заказ.

Ожидаемый результат:

- заказ создается с корректным составом;
- недоступные товары показываются до оплаты;
- согласия обязательны там, где нужны;
- пользователь переходит на оплату с `order_id` или `payment_id`.

## UF-09. Оплата, отмена оплаты и возврат к заказу

Цель: не потерять заказ при успешной, отмененной или неуспешной оплате.

Точка входа:

- `/payment`
- `/orders/:id`

Шаги:

1. Пользователь открывает оплату.
2. Подтверждает или отменяет оплату.
3. Система обновляет платеж и заказ.
4. Пользователь попадает на страницу заказа.

Ожидаемый результат:

- успешная оплата переводит заказ в следующий статус;
- отмена оплаты не удаляет заказ;
- пользователь видит заказ со статусом "Ожидает оплаты";
- есть действия "Оплатить", "Отменить заказ", "Вернуться в магазин".

## UF-10. Напоминания и дневник здоровья

Цель: создать событие ухода и видеть его в профиле питомца.

Точка входа:

- `/pet-id/:id`
- `/health-diary`

Шаги:

1. Пользователь открывает карточку питомца.
2. Создает напоминание или событие.
3. Заполняет дату, время, тип и название.
4. Сохраняет.

Ожидаемый результат:

- элемент появляется в списке;
- UI не показывает ложную ошибку при успешном ответе API;
- повторное открытие страницы показывает сохраненные данные.

## UF-11. Курсы

Цель: купить или открыть обучающий курс.

Точка входа:

- `/courses`
- `/courses/:id`
- рекомендации из PetID

Шаги:

1. Пользователь открывает каталог курсов.
2. Выбирает курс.
3. Покупает платный курс или записывается на бесплатный.
4. Открывает обучение.
5. Проходит страницы уроков и сохраняет прогресс.

Ожидаемый результат:

- курс доступен после покупки/записи;
- прогресс сохраняется;
- пользователь может вернуться к обучению.

## UF-12. Администрирование

Цель: управлять сервисом через административные интерфейсы.

Точка входа:

- `/admin/`
- `/admin-panel/*`
- `/api/admin/`

Шаги:

1. Staff/superuser входит в систему.
2. Открывает нужную админ-панель.
3. Управляет пользователями, питомцами, товарами, заказами, курсами или аналитикой.

Ожидаемый результат:

- обычный пользователь не имеет доступа;
- staff видит React админ-панель;
- superuser имеет доступ к Django admin;
- запреты доступа возвращают понятный экран, а не пустую страницу.
