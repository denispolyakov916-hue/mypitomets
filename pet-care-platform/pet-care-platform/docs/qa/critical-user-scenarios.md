# Критические пользовательские сценарии

## Назначение

Этот документ фиксирует сценарии, которые должны проходить после каждого крупного изменения. Если любой сценарий из списка не проходит, релиз нельзя считать готовым.

Проверять минимум в:

- Chrome desktop;
- Safari desktop;
- мобильная ширина в devtools;
- production-like сборка frontend.

## CUS-01. Email-регистрация и вход

Предусловия:

- пользователь не авторизован;
- email не занят.

Шаги:

1. Открыть `/register`.
2. Зарегистрироваться через email.
3. Если требуется код, активировать аккаунт.
4. Выйти.
5. Открыть `/login`.
6. Войти тем же email и паролем.

Ожидаемый результат:

- регистрация не крашит страницу;
- вход не крашит страницу;
- нет бесконечного loader;
- ошибки показаны понятным текстом;
- пароль не выводится в консоль;
- Safari не показывает неожиданные системные окна из-за некорректной формы.

## CUS-02. Телефонная авторизация

Предусловия:

- доступен backend-режим отправки или тестовой проверки SMS-кода.

Шаги:

1. Открыть `/login`.
2. Переключиться на телефон.
3. Ввести номер.
4. Получить код.
5. Ввести код.
6. Повторить отправку кода после cooldown.

Ожидаемый результат:

- номер форматируется удобно;
- API получает нормализованный номер;
- код отправляется;
- повторная отправка работает;
- неверный код показывает ошибку без краша.

## CUS-03. Создание питомца

Предусловия:

- пользователь авторизован.

Шаги:

1. Открыть `/pet-id`.
2. Добавить питомца.
3. Выбрать собаку.
4. Заполнить имя, породу собаки, возраст/дату рождения, вес, пол, активность.
5. Сохранить.
6. Открыть карточку созданного питомца.

Ожидаемый результат:

- питомец появился в списке;
- данные в карточке соответствуют введенным;
- собачья анкета не показывает кошачьи породы;
- дата рождения не сдвинулась;
- вес сохранился корректно.

## CUS-04. Создание кошки

Шаги:

1. Открыть `/pet-id`.
2. Добавить питомца.
3. Выбрать кошку.
4. Заполнить имя, породу кошки, возраст/дату рождения, вес, пол.
5. Сохранить.

Ожидаемый результат:

- кошачья анкета не показывает собачьи породы;
- питомец сохраняется;
- профиль открывается без ошибки.

## CUS-05. Редактирование питомца

Предусловия:

- есть созданный питомец.

Шаги:

1. Открыть карточку питомца.
2. Нажать "Редактировать профиль".
3. Изменить активность.
4. Изменить вес.
5. Сохранить.
6. Закрыть карточку.
7. Открыть карточку снова.

Ожидаемый результат:

- изменения сохранены в backend;
- UI обновился без ручной перезагрузки;
- после повторного открытия значения не откатываются;
- при ошибке API изменения не показываются как сохраненные.

## CUS-06. Анонимная анкета питания с регистрацией

Предусловия:

- пользователь не авторизован;
- localStorage очищен от старых черновиков.

Шаги:

1. Открыть `/start`.
2. Пройти анкету питомца.
3. Получить рекомендации.
4. Нажать "Добавить рацион в корзину".
5. Зарегистрироваться.
6. Дождаться возврата в сценарий.

Ожидаемый результат:

- данные анкеты не потерялись;
- нет бесконечной загрузки;
- выбранный рацион добавлен в корзину или сохранен согласно выбранному действию;
- если создается питомец/черновик, он создается один раз;
- пользователь видит понятный следующий шаг.

## CUS-07. Подбор питания для существующего питомца

Предусловия:

- пользователь авторизован;
- в аккаунте есть минимум один питомец.

Шаги:

1. Открыть подбор питания с главной.
2. Выбрать существующего питомца.
3. Получить рекомендации.
4. Переключить период 14/30 дней.

Ожидаемый результат:

- не требуется заново вводить известные данные;
- сумма и количество пересчитываются при смене периода;
- рекомендации не падают при неполном профиле.

## CUS-08. Магазин и фильтры

Шаги:

1. Открыть `/shop`.
2. Выбрать категорию "Корма и добавки".
3. Выбрать категорию "Игрушки".
4. Выбрать категорию "Аксессуары".
5. Выбрать категорию "Уход и гигиена".
6. Перезагрузить страницу после выбора фильтра.

Ожидаемый результат:

- каждая категория кликабельна;
- URL содержит фильтр;
- после перезагрузки фильтр сохраняется;
- список товаров соответствует фильтру или показывает корректный empty state.

## CUS-09. Корзина

Предусловия:

- пользователь авторизован;
- в магазине есть доступный товар.

Шаги:

1. Добавить товар в корзину.
2. Открыть `/cart`.
3. Увеличить количество.
4. Уменьшить количество.
5. Удалить товар.

Ожидаемый результат:

- итоговая сумма меняется корректно;
- нет полного мерцания страницы при изменении количества;
- состояние корзины синхронизировано с backend.

## CUS-10. Checkout и создание заказа

Предусловия:

- пользователь авторизован;
- в корзине есть товар.

Шаги:

1. Открыть корзину.
2. Перейти в checkout.
3. Заполнить данные доставки.
4. Выбрать способ оплаты.
5. Создать заказ.

Ожидаемый результат:

- заказ создан;
- пользователь попал на оплату;
- состав заказа соответствует выбранным товарам;
- недостающие согласия блокируют продолжение.

## CUS-11. Отмена оплаты

Предусловия:

- есть заказ в статусе ожидания оплаты.

Шаги:

1. Открыть страницу оплаты.
2. Отменить оплату или смоделировать сбой.
3. Перейти к заказу.

Ожидаемый результат:

- заказ не удален;
- корзина не очищена без объяснения;
- пользователь видит статус "Ожидает оплаты";
- доступны действия "Оплатить" и "Отменить заказ".

## CUS-12. Напоминание питомца

Предусловия:

- пользователь авторизован;
- есть питомец.

Шаги:

1. Открыть карточку питомца.
2. Создать напоминание.
3. Указать название, тип, дату и время.
4. Сохранить.
5. Перезагрузить страницу.

Ожидаемый результат:

- напоминание создано;
- не показана ложная ошибка;
- после перезагрузки напоминание видно;
- пустые обязательные поля валидируются.

## CUS-13. Дневник здоровья

Шаги:

1. Открыть `/health-diary`.
2. Выбрать питомца.
3. Создать событие или запись.
4. Сохранить.

Ожидаемый результат:

- запись сохраняется;
- запись связана с правильным питомцем;
- пользователь видит запись после обновления страницы.

## CUS-14. Курсы

Шаги:

1. Открыть `/courses`.
2. Открыть карточку курса.
3. Добавить платный курс в корзину или записаться на бесплатный.
4. Открыть обучение.

Ожидаемый результат:

- курс доступен после покупки/записи;
- страница обучения открывается;
- прогресс не теряется.

## CUS-15. Админ-панель

Предусловия:

- есть обычный пользователь;
- есть staff/superuser пользователь.

Шаги:

1. Обычным пользователем открыть `/admin-panel/`.
2. Staff/superuser открыть `/admin-panel/`.
3. Superuser открыть `/admin/`.

Ожидаемый результат:

- обычный пользователь получает запрет доступа;
- staff/superuser видит React админ-панель;
- superuser имеет доступ к Django admin;
- запрет доступа отображается понятным экраном.

## Минимальный smoke перед релизом

Перед релизом выполнить:

```bash
cd frontend
npm run build
npm run lint
npm run test:run
```

Если изменялся backend:

```bash
cd backend
python manage.py test
```

Если менялись справочники:

```bash
cd backend
python manage.py check
```

## Что считать блокером

- Краш страницы.
- Бесконечная загрузка.
- Потеря данных после регистрации.
- Ошибка оплаты или заказа без понятного восстановления.
- Невозможность создать или открыть PetID.
- Невозможность войти в аккаунт.
- Утечка пароля, токена или персональных данных в консоль.
