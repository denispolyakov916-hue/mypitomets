# Access Control

Дата ревизии: 2026-06-30

## Назначение

Этот документ описывает текущую модель ролей, доступов и административных зон Питомец+. Он нужен для разработки, QA и безопасного управления проектом.

## Уровни доступа

| Уровень | Кто это | Основной доступ |
|---------|---------|-----------------|
| Гость | Неавторизованный посетитель | Публичные страницы, магазин, курсы, породы, анонимная анкета питания. |
| Пользователь | Авторизованный владелец аккаунта | PetID, корзина, checkout, заказы, профиль, дневник, напоминания, избранное. |
| Course creator | Автор курсов | Доступ к части курсовой админки и редактору курсов. |
| Staff/admin | Администратор | React admin panel, admin API, управление данными. |
| Superuser | Полный администратор Django | Django admin и полный административный доступ. |

## Модель пользователя

Файл:

```text
backend/apps/users/models.py
```

Основные поля:

| Поле | Назначение |
|------|------------|
| `role` | Логическая роль: `user`, `course_creator`, `admin`. |
| `is_staff` | Django/DRF staff-доступ. |
| `is_superuser` | Полный Django-доступ. |
| `is_active` | Активность аккаунта. |
| `is_activated` | Активация через email. |
| `email_verified` | Подтверждение email в профиле. |
| `phone_verified` | Подтверждение телефона. |

Текущая особенность:

```text
User.save() синхронизирует is_staff и is_superuser с role == admin.
```

Это означает:

- `role=admin` делает пользователя staff и superuser;
- `role=course_creator` сам по себе не делает пользователя staff;
- любые ручные изменения `is_staff` могут быть перезаписаны при сохранении пользователя, если роль не `admin`.

## Frontend-доступы

### PrivateRoute

Защищает пользовательские маршруты:

- `/pet-id`;
- `/cart`;
- `/checkout`;
- `/orders`;
- `/profile`;
- `/settings`;
- `/health-diary`;
- обучение и другие приватные страницы.

### AdminRoute

Защищает `/admin-panel/*`.

Frontend считает доступ разрешенным, если:

```text
user.is_staff || user.is_superuser || user.role === 'course_creator'
```

Это означает, что course creator может попасть в React admin shell. Но backend API все равно должен ограничить, какие данные ему доступны.

## Backend permissions

Глобальный DRF default:

```text
DEFAULT_PERMISSION_CLASSES = AllowAny
```

Поэтому каждый защищенный endpoint обязан явно объявлять permission.

Типичные permissions:

| Permission | Где используется |
|------------|------------------|
| `AllowAny` | Публичные справочники, публичные страницы, регистрация, вход. |
| `IsAuthenticated` | Пользовательские данные, питомцы, корзина, заказы, платежи, напоминания. |
| `IsAdminUser` | Большинство admin API. |
| Кастомные permissions | Владение объектом, автор курса, course creator. |

## Публичные зоны

Публичными могут быть:

- регистрация;
- вход;
- активация;
- восстановление пароля;
- публичный магазин;
- публичный каталог курсов;
- публичные страницы пород;
- часть справочников пород и питания;
- анонимная анкета питания.

Публичность endpoint-а не означает, что он может отдавать персональные данные.

## Пользовательские зоны

Требуют авторизации:

- список питомцев;
- создание и редактирование питомца;
- карточка питомца;
- анализ питомца;
- напоминания;
- дневник здоровья;
- корзина пользователя;
- checkout;
- платежи;
- история заказов;
- приобретенные курсы;
- профиль пользователя.

Правило: пользователь должен видеть только свои данные, если он не staff/superuser.

## Административные зоны

| Зона | URL | Доступ |
|------|-----|--------|
| Django admin | `/admin/` | `is_staff`, для полного доступа `is_superuser` и Django permissions. |
| React admin panel | `/admin-panel/*` | Frontend: staff/superuser/course_creator. |
| Admin API | `/api/admin/*` | Backend: в основном `IsAdminUser`, часть курсовых операций может допускать course creator. |

Подробности: `docs/modules/13-admin-panel/admin-access-guide.md`.

## Course creator

`course_creator` - отдельная роль для работы с курсами.

Текущее поведение:

- frontend допускает `course_creator` в `/admin-panel`;
- default redirect ведет course creator на `/admin-panel/courses`;
- часть backend-логики допускает course creator для курсов;
- большинство admin API требует `IsAdminUser`.

Риск: если UI покажет course creator разделы, backend может вернуть 403. Это нормально для защиты, но плохо для UX. UI должен скрывать недоступные разделы по роли.

## Владение объектами

Для пользовательских данных должны действовать правила:

- питомец принадлежит владельцу;
- напоминание связано с питомцем владельца;
- заказ принадлежит пользователю;
- платеж принадлежит пользователю;
- курс доступен пользователю после покупки/записи;
- отзыв можно создавать только при наличии права на отзыв.

При нарушении доступа backend должен возвращать 403 или 404, а не 500.

## Токены

Используется JWT:

- access token lifetime: 1 day;
- refresh token lifetime: 7 days;
- frontend хранит access token в localStorage;
- API принимает `Authorization: Bearer <token>`.

Риски localStorage:

- токен доступен JavaScript-коду;
- XSS может привести к краже токена;
- нельзя логировать токены или передавать их в query string.

Рекомендация для будущей версии: рассмотреть httpOnly cookie-flow для refresh/access или усилить XSS-защиту и CSP.

## Секреты

Нельзя хранить в git:

- `DJANGO_SECRET_KEY`;
- `DB_PASSWORD`;
- SMTP пароль;
- SMSC пароль;
- S3 access/secret key;
- production токены;
- реальные админские пароли.

Шаблоны разрешены только с плейсхолдерами:

- `.env.example`;
- `.env.production.example`.

## Логи

В логи не должны попадать:

- пароли;
- access/refresh token;
- SMS/activation code;
- payment payload с чувствительными данными;
- полные персональные данные без необходимости.

Особенно проверить:

- frontend console logs;
- backend request logging;
- payment logs;
- auth logs.

## Чеклист для нового endpoint

Перед merge нового backend endpoint:

- [ ] Endpoint явно объявляет permission.
- [ ] Проверяется владелец объекта.
- [ ] Ошибка доступа возвращает 401/403/404, не 500.
- [ ] Нет персональных данных в публичном ответе.
- [ ] Нет секретов и токенов в логах.
- [ ] Есть тест на доступ анонима.
- [ ] Есть тест на доступ чужого пользователя.
- [ ] Если endpoint admin-only, он не доступен обычному пользователю.

## Чеклист для нового frontend route

- [ ] Публичный route не показывает приватные данные.
- [ ] Приватный route завернут в `PrivateRoute`.
- [ ] Admin route завернут в `AdminRoute`.
- [ ] UI скрывает действия, недоступные по роли.
- [ ] Backend все равно проверяет доступ независимо от UI.

## Известные зоны риска

| Риск | Почему важно | Что сделать |
|------|--------------|-------------|
| `AllowAny` как глобальный default | Новый endpoint может случайно стать публичным. | Требовать явный permission в code review. |
| `course_creator` в frontend admin shell | Frontend может показать лишние разделы. | Ограничить меню и API по роли. |
| localStorage token | Риск XSS-кражи токена. | Усилить CSP, убрать небезопасные HTML-вставки, рассмотреть cookie-flow. |
| Логи auth-запросов | Можно случайно вывести пароль/токен. | Убрать debug logs с payload. |
| Admin role sync | `User.save()` перезаписывает staff/superuser по role. | Документировать и покрыть тестом. |
