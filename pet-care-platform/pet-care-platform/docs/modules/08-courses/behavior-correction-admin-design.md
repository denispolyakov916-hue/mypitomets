# Админка курсов коррекции поведения

## Цель

Сделать в существующей админке отдельный управляемый сценарий для курсов коррекции поведения, чтобы администратор или автор курса мог:

- создать курс по конкретной поведенческой проблеме;
- заполнить все поля, которые нужны для подбора курса питомцу;
- собрать программу курса из модулей, уроков и блоков;
- проверить безопасность курса перед публикацией;
- видеть пользователей и питомцев, которые проходят курс;
- понимать прогресс, проблемные места и результаты курса.

Курсы коррекции поведения не должны быть отдельным вторым сервисом. Они должны расширять текущий модуль курсов и использовать уже существующие сущности `Course`, `CourseModule`, `CoursePage`, `ContentBlock`, `UserCourse`, `UserCourseProgress`, пользователей и питомцев.

## Текущая база проекта

В проекте уже есть основа, на которую надо опираться:

- `backend/apps/training/models.py`: модель `Course` уже содержит категорию, подкатегорию, автора, статус, цену, тип питомца и поля персонализации.
- `Course.category` уже поддерживает значение `behavior`.
- `Course.subcategory` уже поддерживает поведенческие варианты `behavior_problems`, `aggression`, `anxiety`.
- `Course` уже умеет хранить поля подбора: `recommended_behavior_types`, `recommended_activity_levels`, `recommended_social_levels`, `min_training_experience`, `compatible_health_issues`, `addresses_special_needs`, `suitable_activities`, `addresses_behavioral_problems`.
- `Course.is_compatible_with_pet(pet)` уже считает совместимость курса с питомцем.
- Конструктор курса уже построен на структуре `Course -> CourseModule -> CoursePage -> ContentBlock`.
- В админке уже есть список курсов, форма курса, создание черновика и редактор с вкладками "Настройки" и "Конструктор".

Главная проблема текущего состояния: часть поведенческих полей есть в модели, но не выведена полноценно в админку и не проходит через `AdminCourseViewSet`. Поэтому автор курса не может руками заполнить все данные, которые потом нужны подбору.

## Роли

### Администратор

Полный доступ:

- видит все курсы коррекции поведения;
- создает, редактирует, публикует, архивирует и удаляет курсы;
- назначает автора или специалиста;
- видит пользователей, питомцев, прогресс и обращения по курсам;
- может снять курс с публикации при рисках.

### Автор курса

Роль уже есть как `course_creator`.

Доступ:

- видит только свои курсы;
- создает черновики;
- редактирует паспорт, программу и материалы своих курсов;
- отправляет курс на проверку;
- не публикует курс напрямую, если включена модерация.

### Редактор или проверяющий

Можно добавить позже отдельной ролью. В MVP эту роль выполняет администратор.

Доступ:

- проверяет курс перед публикацией;
- оставляет замечания;
- утверждает безопасность и качество материалов.

## Жизненный цикл курса

Текущие статусы `draft`, `published`, `archived` нужно расширить логикой проверки:

1. `draft`: черновик, доступен автору и администратору.
2. `review`: курс отправлен на проверку, автор не может публиковать сам.
3. `changes_requested`: проверяющий вернул курс на доработку.
4. `published`: курс виден пользователям и участвует в подборе.
5. `archived`: курс скрыт, но история прохождений сохраняется.

Если не расширять enum на первом этапе, можно временно оставить `draft/published/archived`, а состояние проверки хранить отдельным полем `review_status`. Но лучше сразу сделать явный статусный процесс, потому что для коррекции поведения важна безопасность.

## Основные экраны

### 1. Список курсов коррекции поведения

Маршрут: `/admin-panel/courses?type=behavior-correction` или отдельный пункт меню "Коррекция поведения".

На экране нужны:

- поиск по названию, проблеме, автору;
- фильтр по питомцу: собаки, кошки, все;
- фильтр по проблеме: агрессия, тревожность, страхи, туалет, лай/мяуканье, разрушительное поведение, поводок, социализация;
- фильтр по уровню риска;
- фильтр по статусу;
- фильтр "не заполнены поля подбора";
- фильтр "есть блокирующие ошибки публикации";
- быстрый переход в карточку курса;
- действие "Создать курс коррекции".

В таблице показывать:

- название;
- питомец;
- основная проблема;
- уровень сложности;
- риск;
- статус;
- заполненность паспорта;
- количество уроков;
- автор;
- дата обновления.

### 2. Создание курса коррекции

Текущую модалку создания курса нужно расширить режимом "Курс коррекции поведения".

При выборе этого режима система должна автоматически ставить:

- `category = behavior`;
- `format_type = mixed`;
- `status = draft`;
- `is_active = false`;
- автора текущим пользователем;
- обязательное заполнение проблемы поведения.

Минимальные поля при создании:

- название курса;
- для кого курс: собака, кошка, все;
- основная проблема;
- уровень сложности;
- автор или специалист;
- примерная длительность прохождения.

### 3. Паспорт коррекции

Это главная новая вкладка в редакторе курса.

Поля:

- основная проблема поведения;
- дополнительные проблемы;
- видимые симптомы;
- целевое состояние после курса;
- для каких питомцев подходит;
- для каких питомцев не подходит;
- минимальный возраст питомца;
- максимальный возраст питомца;
- ограничения по здоровью;
- ограничения по породе или размеру;
- требуемый опыт владельца;
- сколько времени владелец должен уделять в день;
- какие предметы нужны;
- когда нужен очный специалист;
- когда нужен ветеринар;
- что считается успешным результатом;
- как пользователь должен фиксировать прогресс.

Поля паспорта должны быть не просто текстом. Там, где возможно, нужны списки и теги, чтобы подбор мог работать по данным, а не по свободному описанию.

### 4. Подбор под питомца

Эта вкладка связывает курс с анкетой питомца.

В MVP она должна редактировать уже существующие поля `Course`:

- `recommended_behavior_types`;
- `recommended_activity_levels`;
- `recommended_social_levels`;
- `min_training_experience`;
- `compatible_health_issues`;
- `addresses_special_needs`;
- `suitable_activities`;
- `addresses_behavioral_problems`.

Дополнительно нужны поля:

- `excluded_behavioral_problems`: проблемы, при которых курс нельзя рекомендовать;
- `excluded_health_issues`: состояния здоровья, при которых курс нельзя рекомендовать;
- `min_age_months`;
- `max_age_months`;
- `risk_level`: low, medium, high;
- `requires_specialist_supervision`;
- `requires_vet_clearance`.

При сохранении админка должна показывать, какие данные курса будут участвовать в подборе.

### 5. Безопасность и противопоказания

Отдельная вкладка или блок в паспорте.

Поля:

- предупреждение перед стартом курса;
- красные флаги, при которых владельцу нельзя выполнять курс самостоятельно;
- противопоказания;
- обязательные условия безопасности;
- что делать, если поведение ухудшилось;
- контакты или инструкция для обращения к специалисту;
- чек-лист перед публикацией.

Публикация курса коррекции должна блокироваться, если:

- не выбрана основная проблема;
- не заполнены противопоказания или явно не указано "противопоказаний нет";
- не заполнены красные флаги;
- нет хотя бы одного урока;
- нет блока с вводной безопасностью;
- не заполнены поля подбора под питомца;
- курс высокого риска пытается опубликовать не администратор.

### 6. Конструктор программы

Использовать текущую структуру:

- модуль;
- страница/урок;
- блок контента.

Для курсов коррекции поведения нужны шаблоны страниц:

- вводная безопасность;
- диагностика проблемы;
- упражнение;
- домашнее задание;
- чек-лист выполнения;
- дневник наблюдений;
- разбор типичных ошибок;
- контроль прогресса;
- финальная проверка результата.

Нужны шаблоны блоков:

- текстовая инструкция;
- видео;
- чек-лист;
- домашнее задание;
- вопрос владельцу;
- шкала оценки поведения;
- таймер упражнения;
- загрузка фото или видео отчета;
- предупреждение безопасности;
- прогресс-трекер.

### 7. Ученики и питомцы курса

Новая вкладка в карточке курса.

Показывать:

- кто купил или получил доступ к курсу;
- какого питомца привязали к прохождению;
- возраст, вид, порода, ключевые проблемы питомца;
- процент прохождения;
- последняя активность;
- проблемные уроки;
- комментарии или вопросы пользователя;
- возвраты, если они есть.

Действия:

- открыть карточку пользователя;
- открыть карточку питомца;
- посмотреть прогресс;
- вручную выдать доступ;
- отозвать доступ;
- отметить пользователя как требующего внимания специалиста.

### 8. Аналитика курса

Для каждого курса показывать:

- просмотры карточки;
- покупки или записи;
- старт курса;
- завершения;
- средний прогресс;
- уроки, на которых чаще всего останавливаются;
- вопросы пользователей;
- возвраты;
- выручку;
- средний срок прохождения;
- долю пользователей, дошедших до результата.

В MVP можно показывать только данные, которые уже есть в базе. Фантомные продажи, выручку или результаты подставлять нельзя.

```python
contraindications = models.JSONField(default=list, blank=True)

review_status = models.CharField(max_length=30, default='not_submitted', db_index=True)
review_notes = models.TextField(blank=True, null=True)
```

### Почему не отдельная модель в MVP

Отдельная модель `BehaviorCorrectionCourseProfile` возможна, но для MVP лучше расширить `Course`, потому что:

- подбор уже работает через `Course`;
- админский список уже работает с `Course`;
- публикация уже работает через `Course`;
- каталог курсов уже берет данные из `Course`;
- меньше риск сломать текущую витрину курсов.

Отдельную модель имеет смысл выделить позже, если полей станет сильно больше или появятся разные версии методик.

## API

### Расширить `/api/admin/courses/`

`AdminCourseViewSet._serialize_course()` должен отдавать:

- `course_type`;
- все поля паспорта коррекции;
- все поля подбора;
- `risk_level`;
- `review_status`;
- `review_notes`;
- `students_count`;
- `pets_count`;
- `completion_rate`, если есть данные.

`AdminCourseViewSet._do_update()` должен принимать эти поля и сохранять их.

### Новые действия

Добавить действия:

- `POST /api/admin/courses/{id}/submit-for-review/`;
- `POST /api/admin/courses/{id}/request-changes/`;
- `POST /api/admin/courses/{id}/approve/`;
- `GET /api/admin/courses/{id}/publish-check/`;
- `GET /api/admin/courses/{id}/students/`;
- `GET /api/admin/courses/{id}/analytics/`.

### Проверка публикации

`publish-check` должен возвращать:

```json
{
  "can_publish": false,
  "blocking_errors": [
    "Не заполнены противопоказания",
    "Нет вводного блока безопасности"
  ],
  "warnings": [
    "Не указано рекомендованное время занятий в день"
  ]
}
```

Кнопка публикации в админке должна использовать эту проверку до публикации.

## Подбор курса пользователю

Курс коррекции поведения должен участвовать в подборе только если:

- `course_type = behavior_correction`;
- `status = published`;
- `is_active = true`;
- питомец подходит по виду;
- проблема питомца пересекается с `addresses_behavioral_problems` или `correction_problem_tags`;
- проблема питомца не входит в `excluded_behavioral_problems`;
- проблемы здоровья не входят в `excluded_health_issues`;
- возраст питомца входит в диапазон, если диапазон указан.

Для высокого риска:

- если `requires_specialist_supervision = true`, показывать курс с пометкой о необходимости специалиста;
- если `requires_vet_clearance = true`, показывать предупреждение о консультации с ветеринаром;
- если у питомца есть красный флаг, курс не должен рекомендоваться как самостоятельное решение.

## MVP

В первый рабочий этап нужно сделать:

1. Поле `course_type`.
2. Поля паспорта коррекции и безопасности в `Course`.
3. Миграцию базы.
4. Расширение `AdminCourseViewSet`.
5. Режим создания курса коррекции.
6. Новые вкладки "Паспорт коррекции", "Подбор под питомца", "Безопасность".
7. Проверку публикации.
8. Участие курса в подборе только после публикации.
9. Отображение реальных учеников и питомцев без фантомных данных.

## Второй этап

1. Отправка на проверку и замечания проверяющего.
2. Аналитика прохождения курса.
3. Дневник наблюдений и домашние задания.
4. Загрузка пользователем фото или видео отчета.
5. Пометка "нужен специалист".
6. Шаблоны поведенческих курсов.
7. Сравнение эффективности курсов по проблемам.

## Что не делать

- Не создавать фантомные продажи, учеников, выручку или прогресс.
- Не делать отдельную независимую админку, если можно расширить текущую.
- Не публиковать курс коррекции без противопоказаний и красных флагов.
- Не рекомендовать курс высокого риска как самостоятельное решение без предупреждений.
- Не хранить ключевые поля подбора только в свободном тексте.
- Не ломать обычные курсы: изменения должны быть совместимы с `course_type = general`.

## Критерии приемки

Курс коррекции поведения считается готовым к работе, если:

- администратор может создать курс коррекции из админки;
- автор курса видит и редактирует только свои курсы;
- в курсе можно заполнить паспорт коррекции;
- в курсе можно заполнить правила подбора под питомца;
- публикация блокируется при незаполненных обязательных полях безопасности;
- опубликованный курс появляется в каталоге и подборе;
- неопубликованный курс не виден пользователям;
- курс не рекомендует себя питомцам с противопоказаниями;
- в карточке курса видны реальные пользователи и питомцы, проходящие курс;
- в аналитике нет выдуманных данных;
- обычные курсы продолжают работать как раньше.

## Тестовые сценарии

### Создание курса

1. Администратор открывает список курсов.
2. Нажимает "Создать курс".
3. Выбирает "Коррекция поведения".
4. Заполняет название, питомца и проблему.
5. Система создает черновик с `course_type = behavior_correction`, `category = behavior`, `is_active = false`.

### Проверка публикации

1. Администратор открывает черновик курса коррекции.
2. Не заполняет противопоказания.
3. Нажимает "Опубликовать".
4. Система показывает блокирующую ошибку и не публикует курс.

### Участие в подборе

1. У питомца в анкете указана проблема поведения.
2. Есть опубликованный курс с такой же проблемой в `addresses_behavioral_problems`.
3. У питомца нет противопоказаний.
4. Курс попадает в рекомендации.

### Исключение из подбора

1. У питомца есть проблема или состояние из `excluded_behavioral_problems` или `excluded_health_issues`.
2. Система не рекомендует курс как самостоятельное решение.
3. Пользователь видит предупреждение или рекомендацию обратиться к специалисту, если это предусмотрено логикой сервиса.
