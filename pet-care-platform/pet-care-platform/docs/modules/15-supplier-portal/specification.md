# Кабинет поставщика Динозаврик - Техническое задание

## 1. Назначение

Кабинет поставщика нужен, чтобы подрядчик/поставщик Динозаврик мог самостоятельно поддерживать ассортимент кормов, фасовки, цены, наличие и обязательные данные для подбора питания, а владелец платформы Питомец+ мог вручную проверять и утверждать эти данные перед публикацией в магазине и персональном подборе.

Кабинет также должен показывать поставщику коммерческую аналитику по его товарам: продажи, выручку, количество проданных единиц, возвраты, динамику по периодам и эффективность ассортимента.

Главный принцип: поставщик может создавать и исправлять данные, но не может напрямую включать корм в подбор. В подбор попадают только вручную утвержденные владельцем платформы рецепты.

## 2. Цели

1. Убрать зависимость от ручной правки JSON-файла Динозаврика как единственного способа обновлять ассортимент.
2. Дать поставщику понятный интерфейс для заполнения всех полей, необходимых магазину и подбору.
3. Разделить публикацию в магазин и разрешение для подбора.
4. Сохранить ручной контроль владельца платформы над медицински и питательно значимыми полями.
5. Дать поставщику прозрачную аналитику продаж, выручки и возвратов по его товарам.
6. Использовать JSON-фид как источник импорта, сверки и обновления цен/остатков, а не как единственный источник истины.

## 3. Область работ

### Входит в MVP

1. Роль поставщика и доступ только к своим данным.
2. Список товаров/кормов поставщика.
3. Создание и редактирование черновика корма.
4. Создание и редактирование фасовок/SKU.
5. Валидация полноты данных для магазина и подбора.
6. Отправка корма на модерацию.
7. Админская модерация владельцем платформы.
8. Публикация товара в магазин.
9. Разрешение рецепта в подбор.
10. Дашборд поставщика с продажами, выручкой, количеством, возвратами.
11. Фильтры аналитики по периоду, бренду, товару, SKU, статусу заказа.
12. Экспорт аналитики в CSV/XLSX.

### Не входит в MVP

1. Автоматическая публикация в подбор без проверки владельца платформы.
2. Финансовые взаиморасчеты и акты агентского вознаграждения.
3. Полноценный EDI-документооборот.
4. Редактирование заказов поставщиком.
5. Управление пользовательскими данными покупателей поставщиком.
6. Обработка платежных возвратов поставщиком.

## 4. Участники и роли

### Владелец платформы / администратор

Роль: `admin`.

Права:

1. Видит всех поставщиков.
2. Видит все товары, рецепты, фасовки, заказы и возвраты.
3. Проверяет и утверждает черновики поставщика.
4. Разрешает или запрещает публикацию в магазин.
5. Разрешает или запрещает участие в подборе.
6. Может отклонить данные с комментарием.
7. Может исправить любые поля вручную.
8. Может создавать пользователей поставщика и привязывать их к поставщику.

### Менеджер поставщика

Новая роль: `supplier_manager`.

Права:

1. Видит только своего поставщика.
2. Видит только свои товары, рецепты, офферы, SKU и аналитику.
3. Создает и редактирует черновики.
4. Отправляет данные на модерацию.
5. Видит комментарии модерации.
6. Исправляет отклоненные данные.
7. Видит агрегированные заказы, продажи, возвраты и выручку по своим товарам.
8. Не видит персональные данные покупателей, кроме обезличенных агрегатов.
9. Не может включать `manual_verified`, `is_recommendable` и финальную публикацию в подбор.

### Аналитик поставщика

Новая роль: `supplier_analyst`.

Права:

1. Только чтение.
2. Видит дашборды, отчеты, продажи, возвраты и выгрузки.
3. Не может редактировать ассортимент.

### Редактор поставщика

Новая роль: `supplier_editor`.

Права:

1. Создает и редактирует черновики кормов и фасовок.
2. Отправляет на модерацию.
3. Не видит финансовые дашборды, если доступ не выдан отдельно.

## 5. Связь с текущими сущностями проекта

В проекте уже есть базовые сущности, которые нужно использовать:

1. `FoodRecipe` - нормализованный рецепт корма, источник истины для подбора.
2. `Supplier` - поставщик.
3. `SupplierOffer` - фасовка/оффер поставщика: артикул, вес, цена, наличие.
4. `SupplierRawItem` - сырые данные из JSON/фида.
5. `Product` - товар витрины магазина.
6. `ProductSKU` - конкретная фасовка товара в магазине.
7. `Order` - заказ.
8. `OrderItem` - позиция заказа.
9. `Return` - возврат.
10. `User` - пользователь с ролью.

1. Один пользователь может быть привязан к нескольким поставщикам только через отдельные записи.

| Статус | Значение |

## 7. Данные корма в кабинете поставщика

### Блок "Идентичность рецепта"

Поля:

1. Бренд.
2. Название рецепта без фасовки.
3. Линейка.
4. Вид животного: кошка/собака.
5. Тип корма: сухой/влажный/лакомство.
6. Возрастная группа: kitten/puppy/adult/senior/all.
7. Размерная группа породы: mini/small/medium/large/giant/all.
8. Назначения: sterilized, urinary, renal, gi, skin, weight, hepatic, joints, diabetes, hairball, allergy, grain_free.
9. Основной белок.
10. Признаки: для стерилизованных, чувствительное пищеварение, urinary, контроль веса, беззерновой, гипоаллергенный.

### Блок "Питательность"

Поля на 100 г:

1. Ккал.
2. Белок, %.
3. Жир, %.
4. Клетчатка, %.
5. Зола, %.
6. Влажность, %.
7. Кальций, %.
8. Фосфор, %.

Обязательные поля для подбора:

1. Ккал.
2. Белок.
3. Жир.
4. Влажность желательно, но не блокирует MVP, если остальные данные надежны.

### Блок "Состав"

Поля:

1. Состав текстом как на упаковке.
2. Ингредиенты списком в порядке убывания.
3. Аллергены списком.
4. Основной животный белок.
5. Флаг "состав проверен поставщиком".

Правило: если ингредиенты не заполнены или аллергены не извлечены, корм нельзя отправить в подбор.

### Блок "Фасовки и офферы"

Каждая фасовка должна иметь:

1. Артикул поставщика.
2. Название фасовки.
3. Вес в кг.
4. Цена.
5. Старая цена, если есть.
6. Штрихкод, если есть.
7. Наличие.
8. Остаток, если поставщик передает количественный остаток.
9. Агентский процент, если применяется.
10. Фото фасовки, если отличается от основного.

Правило: один `FoodRecipe` может иметь много `SupplierOffer` и много `ProductSKU`.

## 8. Валидация

### Минимум для публикации в магазин

Товар можно публиковать в магазин, если заполнено:

1. Название.
2. Бренд.
3. Вид животного.
4. Категория/группа товара.
5. Минимум одно фото.
6. Минимум одна фасовка.
7. У фасовки есть артикул, вес, цена и наличие.

### Минимум для участия в подборе

Корм можно разрешить в подборе, если заполнено:

1. Вид животного.
2. Тип корма.
3. Возрастная группа.
4. Назначения и флаги, если они заявлены на упаковке.
5. Ккал/100 г.
6. Белок.
7. Жир.
8. Состав.
9. Аллергены.
10. Основной белок.
11. Есть минимум один активный оффер с ценой, весом и наличием.
12. `review_status = manual_verified`.
13. `is_recommendable = true`.

### Блокирующие ошибки

1. Вид животного не определен.
2. Тип корма не определен.
3. Сухой корм размечен как влажный или наоборот.
4. Корм для кошек размечен как корм для собак или наоборот.
5. Отсутствует ккал/100 г.
6. Отсутствует белок или жир.
7. Нет состава.
8. Нет аллергенов при наличии состава.
9. Нет активных фасовок.
10. Вес фасовки не распознан.
11. Цена отсутствует или равна 0.
12. Лечебное назначение не подтверждено названием/упаковкой/составом.

### Предупреждения

1. Нет влажности.
2. Нет золы.
3. Нет кальция/фосфора.
4. Нет штрихкода.
5. Нет второго фото.
6. Название рецепта содержит фасовку.
7. Есть похожий рецепт, возможен дубль.
8. В JSON-фиде цена/наличие отличается от текущей базы.

## 9. Модерация

### Процесс

1. Поставщик создает или исправляет черновик.
2. Система запускает валидацию.
3. Если блокирующих ошибок нет, поставщик нажимает "Отправить на проверку".
4. Администратор открывает заявку.
5. Администратор сравнивает данные с упаковкой, карточкой поставщика или JSON-фидом.
6. Администратор выбирает одно из действий:
   - "Вернуть на исправление".
   - "Отклонить".
   - "Разрешить для магазина".
   - "Разрешить для подбора".
7. При разрешении для магазина создаются/обновляются `Product`, `ProductSKU`, `SupplierOffer`.
8. При разрешении для подбора создается/обновляется `FoodRecipe`, ставятся `review_status = manual_verified`, `is_recommendable = true`, `nutrition_complete = true`, если БЖУ полные.

### Правила безопасности подбора

1. Поставщик не может менять `manual_verified`.
2. Поставщик не может менять `is_recommendable`.
3. Поставщик не может удалять историю модерации.
4. Если поставщик меняет состав, аллергены, ккал, белок, жир, вид животного, тип корма или лечебное назначение, рецепт должен автоматически уходить на повторную модерацию.
5. До повторного утверждения измененная версия не должна влиять на подбор.

## 10. Экраны кабинета поставщика

### 10.1. Вход и выбор поставщика

Путь: `/supplier-panel`.

Функции:

1. Вход через общий JWT-механизм проекта.
2. Проверка роли `supplier_manager`, `supplier_editor`, `supplier_analyst`.
3. Если пользователь привязан к нескольким поставщикам, показывать выбор поставщика.

### 10.2. Дашборд

Путь: `/supplier-panel/dashboard`.

Виджеты:

1. Выручка за период.
2. Количество заказов.
3. Количество проданных единиц.
4. Средний чек по товарам поставщика.
5. Возвраты: количество и сумма.
6. Процент возвратов.
7. Топ-10 товаров по выручке.
8. Топ-10 товаров по количеству.
9. Товары без наличия.
10. Товары, требующие исправлений.
11. Корма, не допущенные в подбор из-за ошибок.
12. Динамика продаж по дням/неделям/месяцам.

Фильтры:

1. Период: сегодня, 7 дней, 30 дней, 90 дней, год, произвольный диапазон.
2. Бренд.
3. Вид животного.
4. Тип корма.
5. Товар.
6. SKU/фасовка.
7. Статус заказа.
8. Статус возврата.

### 10.3. Ассортимент

Путь: `/supplier-panel/products`.

Таблица:

1. Фото.
2. Название рецепта/товара.
3. Бренд.
4. Вид животного.
5. Тип корма.
6. Количество фасовок.
7. Минимальная цена.
8. Наличие.
9. Статус магазина.
10. Статус подбора.
11. Статус модерации.
12. Ошибки валидации.
13. Дата обновления.

Действия:

1. Создать корм.
2. Редактировать.
3. Дублировать.
4. Отправить на проверку.
5. Скрыть из продажи, если право разрешено.
6. Экспортировать список.

### 10.4. Карточка корма

Путь: `/supplier-panel/products/:id`.

Вкладки:

1. Основное.
2. Питательность.
3. Состав и аллергены.
4. Фасовки.
5. Фото.
6. Валидация.
7. История изменений.
8. Комментарии модерации.

На карточке должны быть видны:

1. Поля, которые блокируют публикацию.
2. Поля, которые блокируют подбор.
3. Отличия от последнего JSON-фида.
4. Отличия от текущего утвержденного `FoodRecipe`.
5. Кнопка "Отправить на проверку".

### 10.5. Заказы

Путь: `/supplier-panel/orders`.

Поставщик видит только позиции заказов, в которых есть его товары.

Таблица:

1. ID заказа.
2. Дата заказа.
3. Статус заказа.
4. Товар.
5. SKU/фасовка.
6. Артикул.
7. Количество.
8. Цена за единицу.
9. Сумма позиции.
10. Статус возврата по позиции.

Ограничения:

1. Не показывать email, телефон, адрес и ФИО покупателя в MVP.
2. Поставщик не может менять заказ.
3. Для операционной необходимости можно добавить обезличенный город/регион отдельным этапом.

### 10.6. Возвраты

Путь: `/supplier-panel/returns`.

Таблица:

1. ID возврата.
2. ID заказа.
3. Дата запроса.
4. Товар.
5. SKU/фасовка.
6. Количество.
7. Причина.
8. Статус.
9. Сумма возврата.
10. Комментарий администратора.

Поставщик только читает возвраты. Управляет возвратами администратор платформы.

### 10.7. Аналитика

Путь: `/supplier-panel/analytics`.

Графики:

1. Выручка по дням.
2. Количество заказов по дням.
3. Проданные единицы по дням.
4. Возвраты по дням.
5. Выручка по брендам.
6. Выручка по товарам.
7. Продажи по фасовкам.
8. Конверсия ассортимента: товары в наличии / товары без наличия / товары в подборе / товары заблокированы.

Таблицы:

1. Топ товаров по выручке.
2. Топ товаров по количеству.
3. Товары с высоким процентом возвратов.
4. Товары без продаж.
5. Товары, которые есть в магазине, но не участвуют в подборе.
6. Товары, которые участвуют в подборе, но не имеют активных офферов.

### 10.8. Импорт и сверка JSON

Путь: `/supplier-panel/imports`.

Функции:

1. Видеть историю импортов JSON-фида.
2. Видеть сколько товаров создано, обновлено, не изменилось, упало с ошибкой.
3. Видеть список расхождений:
   - цена изменилась;
   - наличие изменилось;
   - исчезла фасовка;
   - появилась новая фасовка;

## 12. Расчет метрик дашборда

Все метрики поставщика считаются только по позициям заказа, где:

1. `OrderItem.product.supplier = текущий supplier`, или
2. `OrderItem.sku.supplier_offer.supplier = текущий supplier`, или
3. `OrderItem.product.food_recipe.offers.supplier = текущий supplier`, если нужна fallback-связь.

Основной статусный фильтр продаж:

1. В выручку включать заказы со статусами `processing`, `shipped`, `partially_delivered`, `delivered`.
2. Не включать `pending`, `cancelled`, `expired`.
3. Возвраты учитывать отдельно и показывать валовую и чистую выручку.

### Формулы

Валовая выручка:

```text
sum(OrderItem.price * OrderItem.quantity)
```

Количество проданных единиц:

```text
sum(OrderItem.quantity)
```

Количество заказов:

```text
count(distinct Order.id)
```

Средний чек по товарам поставщика:

```text
gross_revenue / orders_count
```

Сумма возвратов:

```text
sum(Return.refund_amount), где Return.order_item относится к supplier
```

Количество возвращенных единиц:

```text
sum(Return.quantity)
```

Процент возвратов по количеству:

```text
returned_units / sold_units * 100
```

Чистая выручка:

```text
gross_revenue - refunded_amount
```

## 14. UI/UX требования

1. Интерфейс должен быть рабочим инструментом, не лендингом.
2. Основной экран - дашборд и таблицы, без декоративных секций.
3. Таблицы должны поддерживать сортировку, фильтры, поиск и пагинацию.
4. Ошибки валидации должны быть видны прямо в карточке корма.
5. Поля, влияющие на подбор, должны быть визуально отделены от коммерческих полей.
6. Нельзя скрывать критичные статусы: "не проверено", "не в подборе", "нет БЖУ", "нет аллергенов".
7. Для аналитики нужны графики и таблицы с drill-down до товара и SKU.
8. На мобильной ширине достаточно просмотра дашборда и таблиц; полноценное редактирование корма оптимизируется под desktop.

## 15. Права доступа и безопасность

1. Все supplier-запросы должны фильтроваться по `SupplierUserAccess`.
2. Нельзя доверять supplier_id из query/body без проверки доступа.
3. Поставщик не должен получать персональные данные покупателей.
4. Поставщик не должен видеть товары других поставщиков.
5. Поставщик не должен видеть закупочные или внутренние бизнес-приоритеты платформы, если они не предназначены для него.
6. Все изменения корма должны писаться в историю.
7. Все действия модерации должны писать `reviewed_by`, дату и комментарий.
8. Экспорт отчетов должен логироваться.

## 16. История изменений

Для каждого черновика и утвержденного рецепта нужна история:

1. Кто изменил.
2. Когда изменил.
3. Какие поля изменил.
4. Старое значение.
5. Новое значение.
6. Источник изменения: supplier UI, admin UI, JSON import, system parser.

Минимальная реализация: JSON-поле `changed_fields` в заявке и отдельная таблица аудита на следующем этапе.

## 17. Интеграция с JSON Динозаврика

JSON-фид должен использоваться в трех сценариях:

1. Первичный импорт ассортимента.
2. Регулярное обновление цен, наличия и фасовок.
3. Сверка расхождений между фидом, черновиками и утвержденной базой.

Правила:

1. Цена и наличие могут обновляться автоматически, если артикул и поставщик однозначно совпадают.
2. Новая фасовка создается как черновик или оффер со статусом "нужна проверка", если не хватает веса/цены/артикула.
3. Изменение состава, БЖУ, назначения, вида животного или типа корма не должно автоматически менять утвержденный рецепт для подбора.
4. Такие изменения должны создавать заявку на модерацию.

## 18. Статусы финальных сущностей

### FoodRecipe

Для подбора использовать только:

```text
is_recommendable = true
review_status in ("manual_verified")
kcal_per_100g is not null
```

`auto_parsed` можно показывать как кандидатов для проверки, но не считать ручным утверждением.

### Product

Для магазина использовать:

```text
status = 1
is_available = true
product_group = "food"
supplier = текущий supplier
```

Для покупки и расчета дней:

```text
ProductSKU.available = true
ProductSKU.status = 1
SupplierOffer.in_stock = true
SupplierOffer.price is not null
SupplierOffer.package_weight_kg is not null
```

## 21. Критерии приемки

1. Поставщик может войти в кабинет и видит только свои данные.
2. Поставщик может создать корм и фасовки.
3. Система показывает, каких данных не хватает для магазина.
4. Система показывает, каких данных не хватает для подбора.
5. Поставщик может отправить корм на проверку только без блокирующих ошибок черновика.
6. Администратор может вернуть заявку на исправление с комментарием.
7. Администратор может разрешить товар в магазин без разрешения в подбор.
8. Администратор может разрешить корм в подбор только после чеклиста ручной проверки.
9. После разрешения в подбор у `FoodRecipe` стоит `manual_verified` и `is_recommendable = true`.
10. Если поставщик меняет критичные поля, корм уходит на повторную модерацию.
11. Дашборд поставщика показывает выручку, заказы, проданные единицы, возвраты и чистую выручку за выбранный период.
12. Поставщик не видит персональные данные покупателей.
13. Поставщик не видит товары и аналитику других поставщиков.
14. CSV/XLSX экспорт содержит только данные текущего поставщика.
15. JSON-импорт показывает историю и расхождения.
