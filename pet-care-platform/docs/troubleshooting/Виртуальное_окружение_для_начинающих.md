## Что такое виртуальное окружение?

### Простая аналогия: Квартира vs Общежитие

Представь, что Python — это большой дом, где живут программы. Каждой программе нужны свои "вещи" — это называется **библиотеки** (или пакеты).

**Без виртуального окружения** — это как жить в общежитии:
- Все программы живут в одной комнате
- Все вещи (библиотеки) лежат в одной куче
- Если одна программа хочет новую версию библиотеки, а другая — старую, начинается хаос!

**С виртуальным окружением** — это как отдельная квартира:
- У каждого проекта своя изолированная "комната"
- Свои библиотеки, которые не мешают другим
- Полный порядок!

### Техническое определение

**Виртуальное окружение (venv)** — это изолированная папка, которая содержит:
- Копию интерпретатора Python
- Свой набор установленных библиотек
- Скрипты для активации/деактивации

```
Твой проект/
├── venv/                    ← Виртуальное окружение (отдельная "квартира")
│   ├── Scripts/             ← Скрипты (Windows)
│   │   ├── python.exe       ← Своя копия Python
│   │   ├── pip.exe          ← Свой установщик пакетов
│   │   └── activate.bat     ← Скрипт активации
│   └── Lib/
│       └── site-packages/   ← Твои библиотеки живут здесь
├── main.py                  ← Твой код
└── requirements.txt         ← Список нужных библиотек
```

## Зачем оно нужно?

### Проблема 1: Конфликт версий

Представь ситуацию:

```
Проект А: нужна библиотека Django версии 3.0
Проект Б: нужна библиотека Django версии 4.0
```

Без виртуального окружения ты можешь установить только ОДНУ версию Django на весь компьютер. Значит, один из проектов сломается!

**С виртуальным окружением:**
```
Проект А/venv/ → Django 3.0 (работает!)
Проект Б/venv/ → Django 4.0 (тоже работает!)
```

### Проблема 2: "У меня работает, а у тебя нет"

Ты написал программу, она работает. Отправил другу — не работает. Почему?

Потому что у тебя установлены одни библиотеки, а у друга — другие (или их вообще нет).

**Решение с venv:**
1. Ты создаёшь файл `requirements.txt` со списком всех библиотек
2. Друг создаёт venv и устанавливает всё из этого списка
3. У вас одинаковое окружение — программа работает у обоих!

### Проблема 3: Засорение системы

Без venv все библиотеки ставятся "глобально" в систему:
- Через год у тебя 500 библиотек
- Ты не помнишь, какая для чего нужна
- Система тормозит, конфликты, баги...

**С venv:**
- Удалил папку `venv` = удалил все библиотеки проекта
- Чистота и порядок!

## Как это работает?

### Что происходит при активации?

Когда ты запускаешь `activate.bat`, происходит магия:

```
БЫЛО (до активации):
Команда "python" → Глобальный Python (C:\Python312\python.exe)
Команда "pip"    → Глобальный pip

СТАЛО (после активации):
Команда "python" → Python из venv (D:\проект\venv\Scripts\python.exe)
Команда "pip"    → pip из venv
```

Технически, скрипт активации просто **меняет переменную PATH**:

```
PATH = D:\проект\venv\Scripts; C:\Python312; C:\Windows\...
         ↑
    Эта папка теперь ПЕРВАЯ в списке поиска!
```

Когда ты пишешь `python`, Windows ищет `python.exe` по папкам из PATH слева направо. Находит первым — `venv\Scripts\python.exe`.

## Когда активировать?

### Правило простое: ВСЕГДА, когда работаешь с проектом

**Активируй виртуальное окружение ПЕРЕД тем, как:**

| Действие | Нужна активация? |
|----------|------------------|
| Запускаешь свой Python-скрипт | ✅ ДА |
| Устанавливаешь библиотеку (`pip install ...`) | ✅ ДА |
| Запускаешь Django-сервер | ✅ ДА |
| Запускаешь тесты | ✅ ДА |
| Смотришь список установленных библиотек | ✅ ДА |
| Просто редактируешь код в редакторе | ❌ Не обязательно* |

*Хотя многие редакторы (VS Code, PyCharm) сами активируют venv.

### Как понять, активировано ли окружение?

Посмотри на командную строку:

```
БЕЗ активации:
C:\Users\Ты\проект>

С активацией:
(venv) C:\Users\Ты\проект>
  ↑
 Вот этот префикс "(venv)" показывает, что окружение активно!
```

### Пример рабочего процесса

```batch
:: 1. Открыл терминал
C:\Users\Ты>

:: 2. Перешёл в папку проекта
C:\Users\Ты> cd D:\pet_develop\Pet_dev\pet-care-platform\backend

:: 3. Активировал виртуальное окружение
D:\...\backend> venv\Scripts\activate.bat

:: 4. Теперь видишь (venv) — можно работать!
(venv) D:\...\backend> python manage.py runserver

:: 5. Всё! Сервер запущен с правильными библиотеками
```

## Когда деактивировать?

### Короткий ответ: Когда закончил работу с проектом

**Деактивируй КОГДА:**

| Ситуация | Нужно деактивировать? |
|----------|----------------------|
| Закончил работу над проектом | ✅ Да (или просто закрой терминал) |
| Переключаешься на другой Python-проект | ✅ Да (потом активируй другой venv) |
| Хочешь использовать глобальный Python | ✅ Да |
| Просто закрываешь терминал | ❌ Не нужно (само деактивируется) |

### Как деактивировать?

```batch
:: Просто напиши:
(venv) D:\...\backend> deactivate

:: Префикс (venv) исчезнет:
D:\...\backend>
```

### Важно понимать!

Виртуальное окружение **активно только в текущем окне терминала**!

```
┌─────────────────────┐    ┌─────────────────────┐
│   Терминал 1        │    │   Терминал 2        │
│                     │    │                     │
│ (venv) активирован  │    │ venv НЕ активирован │
│                     │    │                     │
│ python → venv       │    │ python → глобальный │
└─────────────────────┘    └─────────────────────┘

Активация в одном окне НЕ влияет на другие!
```

## Практические команды

### Создание виртуального окружения

```batch
:: Перейди в папку проекта
cd D:\мой_проект

:: Создай виртуальное окружение (имя "venv" — стандартное)
python -m venv venv

:: Это создаст папку "venv" со всем необходимым
```

### Активация (Windows CMD)

```batch
:: Из папки проекта:
venv\Scripts\activate.bat

:: Или полный путь:
D:\мой_проект\venv\Scripts\activate.bat
```

### Активация (Windows PowerShell)

```powershell
# Из папки проекта:
.\venv\Scripts\Activate.ps1

# Если ошибка "выполнение скриптов запрещено":
Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser
```

### Деактивация

```batch
deactivate
```

### Установка библиотек

```batch
:: Сначала активируй venv, потом:
pip install django
pip install pillow
pip install requests

:: Или все сразу из файла:
pip install -r requirements.txt
```

### Сохранение списка библиотек

```batch
:: Сохранить текущие библиотеки в файл:
pip freeze > requirements.txt

:: Файл будет содержать что-то вроде:
:: Django==4.2.7
:: Pillow==10.1.0
:: requests==2.31.0
```

### Удаление виртуального окружения

```batch
:: Просто удали папку venv:
rmdir /s /q venv

:: Или через проводник — удали папку venv
```

## Частые вопросы

### Q: Нужно ли добавлять папку venv в Git?

**A: НЕТ!** Папка `venv` не должна быть в репозитории. Она:
- Занимает много места (сотни МБ)
- Специфична для твоей системы
- Легко воссоздаётся из `requirements.txt`

Добавь в `.gitignore`:
```
venv/
```

### Q: Что делать, если venv сломался?

**A:** Удали и создай заново:
```batch
rmdir /s /q venv
python -m venv venv
venv\Scripts\activate.bat
pip install -r requirements.txt
```

### Q: Можно ли назвать папку по-другому?

**A:** Да! Но `venv` — стандартное имя, которое понимают все инструменты:
```batch
python -m venv моё_окружение
моё_окружение\Scripts\activate.bat
```

### Q: Почему в проекте "Питомец+" venv уже создан?

**A:** Чтобы тебе не нужно было вручную устанавливать все библиотеки. Проект использует много пакетов (Django, Pillow, и др.), и они уже настроены.

### Q: Что такое `requirements.txt`?

**A:** Это "список покупок" для pip. Содержит все библиотеки с версиями:

```
Django==4.2.7
djangorestframework==3.14.0
Pillow==10.1.0
psycopg2-binary==2.9.9
```

Команда `pip install -r requirements.txt` устанавливает всё из списка.

### Q: Чем venv отличается от virtualenv, conda, poetry?

**A:** Все они делают похожую вещь — изолируют окружения:

| Инструмент | Особенности |
|------------|-------------|
| **venv** | Встроен в Python, простой, достаточен для большинства |
| **virtualenv** | Более старый, чуть больше возможностей |
| **conda** | Для data science, управляет и Python, и системными библиотеками |
| **poetry** | Современный, управляет и зависимостями, и публикацией пакетов |

Для начинающих **venv** — лучший выбор!

Виртуальное окружение — это твой лучший друг при разработке на Python. Оно:

1. **Изолирует** библиотеки разных проектов
2. **Предотвращает** конфликты версий
3. **Упрощает** работу в команде
4. **Поддерживает** чистоту системы

Правило номер один: **Всегда работай в виртуальном окружении!**
